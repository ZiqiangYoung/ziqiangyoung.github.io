Powerdesigner使用心得

发表于 2020-11-21 更新于 2021-05-05 阅读次数： 本文字数： 3.4k 阅读时长 ≈ 3 分钟

数据流图：通过描述数据流来流水线式的描述业务流程，生成的模型也是单纯的图+数据字典，不用来生成代码和数据库，关键是理清业务逻辑，而不是怎么做的规范。

前言

本文只是随笔，正确率堪忧。

有参考价值的总结在文末，请自行判断。

正文

数据流图DFD

  1. 理清业务逻辑，把他当成思维导图或流程图来对待，DFD的本质是BPM，就是需求分析中的一个部分

  2. 生成业务流程文档，可以作为一种需求文档使用

使用powerdesigner绘制DFD：

  1. 和Visio/processon/xmind……一样，还算便于绘图

  2. 对象思想穿插，了解面对对象的人用起来效率不错，项目结构简洁、对象单元复用性较高

  3. 能自动生成文档，包括数据字典、图片、封面、层次、report一键搞定

工具栏/选项卡说明

Toolbox

只展开Standard和DFD栏，其他务必折叠，和DFD不匹配

Entity properties

A|t+Enter展开Properties

输入Name和Code

Comment建议写上这个东西的含义、开发进度版本，手动完成本地版本管理

希望名称中文代码英文方便导出文档，把输入框右侧的等号点击取消即可

StereoType不要改，更改刻板印象只影响图标外形，但会干扰后续设计和排版

Organization是实体从属关系，比如Young属于ZhangClass

Multi-instance多实例，比如学生有多个实例，并不特指一人，此时就勾选。

特指一个人或唯一账号，不要勾选。比如Young只指我一人，root只有唯一账号

Keywords，就是Tag索引法，方便系统庞大时检查和修改，但是具体怎么利用Tag，不明，暂时不管

Definition选项卡，详细说明，一般是打开外部的文档用来详细说明一个单元。绝大多数情况用不上

Object Brower

展开DFD-选中对象拖到打开的Diagram上，就是创建了快捷方式/引用，你创建的所有对象，都存在这里，而不是你创建它的Diagram上，删除对象的最后一个引用/快捷方式时，你可以选则彻底删除对象，还是只删除引用/快捷方式，后者你仍可以从Brower里找到这个对象进行操作，此外，删除引用/快捷方式时，请确保该对象连接的Flow删除了，否则一个Diagram上有两个引用/快捷方式时，你删除一个快捷方式，它上面的Flow会自动指向剩余的那个引用/快捷方式上，造成后续困难

Process properties

大体上和Entity一样，Organization Unit指执行process的单元，比如扫地process是保洁门执行的

Timeout指这个process执行多久还未结束，只是预期，生成文档时要用

Duration指预期这个process多久执行完毕

Composite Process

选中Decomposed process代表这个流程很复杂，要分成几个小步骤来实现

选中及自动生成下一层数据流图，默认情况下不要选择，因为你选了但是没做下层具体内容，模型会报错

Atomic task，原子操作，最简单的操作，所有的复杂操作最后都是由原子操作实现的

NumberID

默认按照创建Diagram位置和该Diagram中Process创建顺序排的

Symbol->Auto Layout自动排版

排版效果不错

Check Model

右键Check Model自动检查模型是否有错误

概念浅谈

事实上在结课之后发现以下很多想法都错了，但是毕竟结课了就用不到powerdesigner了(很现实)，也就懒得重新审视以前的稚嫩观点了，作为漏洞百出的参考算了。

0层数据流图

只有一个process，把你要开发的整个系统封装成这一个系统

注意，对于复杂产品而言，比如说：社交项目包含以下子系统包含通讯、社区、信息中心、文件、市场

规模极其庞大且各子系统之间联系不紧密时，建议拆分项目分别开发子系统

创建一个powerdesigner项目：社交产品

在brower里创建文件夹：通讯、社区、信息中心、文件、市场

不同文件夹里分别需求或设计子系统的DFD和CDM……

几个子系统之间有依赖关系的，老样子拖动引用/快捷方式就可以，

保持数据中台思想，保证信息中心的数据高质量高价值高简洁

把注册、用户信息、有储存价值和高复用性价值的，放在信息中心处理

数据流图对业务模块之间的协作表示的并不好，你想象中一个页面的几个功能实际上都是嵌入进去互相协作的，但如果没有数据上的沟通，数据流图就很容易给读者一种两者好像没有关联的错觉，开发的时候要注意，DFD只是一种BPM，并不是最优BPM，powerdesigner也只是支持DFD，而不是推荐。

1层数据流图

描述子系统(0层)的大致业务流程

1层数据流图务必不要犯思维僵化的错误，书中的各种管理管理，很容易让人迷失，比如说登录模块，就有人着迷，把一个系统在拆分成了几个子系统再分别展开，比如信息修改，1层把0层的通讯系统分成了用户通讯系统和群组通讯系统，2层才展开这个通讯的业务流程，不是的。正确应当是1层描述整体业务的流程，2层是对其的展开详细说明而已，不要搞错了，更不要生硬的学书上一样加入…管理的字样。

1层数据流图的process要求要高度复用，概念上的复用：例如普通用户登录和企业用户登录，只需要创建一个注册process一个登录process，然后拖出两套图形就可以了分别连flow就好，方便看，只有一套图形的话flow会非常乱，两种方法在底层是一致的，不必担心，后面多出来的1、2这种process编号只是区分图标是那一套的，不是两个process不同。

DFD默认方案会自动给Process编号，1.1.1.1这样，编号的顺序只影响排版顺序，不介意的不用管，实在介意的就先绘制完整的流程图图出来然后广度优先遍历操作PowerDesigner好了，还可以手动改NumberID

业务层开发时，需要隐藏数据库细节，所以在一层数据流图时就要注意低耦合原则，就是一个process只针对一张或有联系的多张表：例如群组表就和用户表没有联系，但是用户表和关系表就是有联系的，那么涉及到用户和群组两张表内容的，就应当放在不同的process里，至于外键的约束，DFD里不需要体现，因为DFD只表示数据的流向，而外键的是参考对照，而非数据通过建立外键而流动。

Flow既可以表示process之间的协作关系，又可以承载数据的传递

2层数据流图

二层才开始要有DataStore参与当中

Flow和process、Entity、DataStore不同，后者是可复用的对象，Flow不可以，Flow仅仅在本图层内表示方向和流动，也就是说，你在从高层进入底层时，高层的前后间Flow会消失，只剩下前后衔接的对象，而这时候如果不添加Flow连接这些漂浮的对象，数据流图在底层便断开了，CheckModel时可能报错。而为了防止这个问题，需要添加Flow连接上他们，这时候，上层的Flow和下层的Flow含义重复了，而创建了两个Flow对象，这时候就不要陷入迷茫，Flow的生存域只在当前Diagram，离开了这个Diagram，Flow立刻失去意义，Flow的意义仅仅是图层内的展示！这个对象的价值只在当前Diagram中，离开这个Diagram，Flow就视作不存在

总结

做课程设计的时候因为我是单人成组且自行命题的，弯路走了很多(指3个月的课2.5个月都在画DFD还没画完)

  1. 需求分析很关键，理清楚，然后就尽量不要改需求了，毕竟是维护不如重做系列。
  2. DFD严格画起来很痛苦，所以随意一点就好，千万不要和我一样跟powerdesigner较真，这玩意认真做起来DFD是明显有bug的。
  3. 表设计很简单，仔细一点不要冗余，满足BNCF范式就好了
  4. 至于CDM，前面的需求分析好好做，DFD画的认真点，ER图水到渠成的事情。
  5. LDM、PDM、DD、SQL都是由以上导出再改的，没什么技术含量
  6. Web端写个简单的JavaWeb登录和后台账户管理就行

这样一套流程下来课程设计得分是95，项目文件和报告在我github的powerdesigner仓库里挂着，有兴趣可以去看一下。
